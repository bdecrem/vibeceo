# Agentic To-Do App — Architecture Sketch

## Concept
A to-do app that starts minimal (add/check/delete tasks). Each user has their own "fork" of the app. A built-in chat agent modifies *their* version on request. Admin can promote user-specific features upstream to all users.

## Data Model (Supabase)

**`tasks`** — the actual to-do data
- `id`, `user_id`, `title`, `completed`, `created_at`, `properties` (JSONB — extensible bag for due dates, priority, tags, whatever the agent adds)

**`user_app_config`** — each user's app definition
- `user_id`, `schema` (JSONB — describes which fields/features exist), `ui_config` (JSONB — layout, views, sort/filter prefs), `version`

**`base_app_config`** — the canonical upstream version
- Same shape as `user_app_config`. New users get a copy of this. When admin "merges" a user's feature, this is what gets updated.

## The Agent Loop

1. User types in chat: "add a priority field — high, medium, low"
2. Agent receives: current `user_app_config` + the request
3. Agent outputs: a **diff** to `user_app_config` (new field in schema, UI update to show it) + any migration logic (e.g. backfill existing tasks with `priority: null`)
4. Diff is applied to that user's config. Their app re-renders.

**Key insight:** The agent doesn't generate arbitrary frontend code. It modifies a **declarative config** that a generic renderer interprets. This keeps the agent's job tractable and the output predictable.

## The Renderer

A single React app that reads `user_app_config` and renders accordingly. Think of it as a form/list engine:
- Schema says "there's a `priority` field, type `enum`, options `[high, med, low]`" → renderer shows a dropdown
- UI config says "default sort: priority desc" → renderer sorts
- UI config says "view: kanban, group_by: priority" → renderer switches to columns

You build a finite set of **renderable primitives** (text field, enum, date, checkbox, tags, kanban view, list view, calendar view) and the agent composes them.

## Upstream Merge (Admin Flow)

Admin dashboard shows a feed of agent-applied diffs across users. Admin can:
- **Preview** what a user's config looks like
- **Accept** — merge the diff into `base_app_config`, so new users get it and existing users can opt-in
- **Reject / Ignore**

## Boundaries for the Agent

System prompt includes: *"You can add, modify, or remove task fields, views, filters, sort orders, and UI layout. You work within these primitive types: text, number, date, enum, boolean, tags, url. You can add these view types: list, kanban, calendar. You cannot add features unrelated to task management — no chat, no file storage, no integrations."*

## V1 Scope

Ship with: title (text), completed (boolean), list view, add/check/delete. That's it. Everything else comes from users asking the agent.
